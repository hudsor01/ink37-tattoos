**Part B: Activate after() for audit logging in Server Actions** (per D-22):

For each server action file, replace the `logAudit(...).catch(() => {})` pattern with `after(() => logAudit(...))`:

1. Add import to each action file: `import { after } from 'next/server';`

2. Replace every instance of:
   ```typescript
   logAudit({
     userId: session.user.id,
     action: 'CREATE',
     resource: 'customer',
     resourceId: result.id,
     ip: hdrs.get('x-forwarded-for') ?? 'unknown',
     userAgent: hdrs.get('user-agent') ?? 'unknown',
     metadata: { changes: validated },
   }).catch(() => {});
   ```
   With:
   ```typescript
   after(() =>
     logAudit({
       userId: session.user.id,
       action: 'CREATE',
       resource: 'customer',
       resourceId: result.id,
       ip: hdrs.get('x-forwarded-for') ?? 'unknown',
       userAgent: hdrs.get('user-agent') ?? 'unknown',
       metadata: { changes: validated },
     })
   );
   ```

   The `after()` function runs the callback after the response is sent to the client, so audit logging no longer blocks the response. No `.catch()` needed -- after() handles errors internally.

3. Apply this transformation to ALL action files that use `logAudit`:
   - customer-actions.ts (3 instances: create, update, delete)
   - appointment-actions.ts (read file to find instances)
   - session-actions.ts (read file to find instances)
   - media-actions.ts (read file to find instances)
   - order-actions.ts (read file to find instances)
   - product-actions.ts (read file to find instances)
   - settings-actions.ts (read file to find instances)

4. Capture the `hdrs` variable BEFORE the `after()` call since `headers()` must be called during request processing, not in the after callback. The current code already does this correctly by assigning `const hdrs = await headers()` before the logAudit call.

   Important: The `after()` callback runs after the response, but all the values it references (userId, ip, userAgent, etc.) are captured in the closure when after() is called during request processing. This is safe.
