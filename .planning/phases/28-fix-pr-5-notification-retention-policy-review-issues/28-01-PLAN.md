Create new file `src/lib/security/cron-auth.ts` with two exported functions:

1. `verifyCronAuth(request: Request): boolean` — Per D-01, shared across all 3 cron routes.
   - Reads `CRON_SECRET` from `process.env`
   - If CRON_SECRET is not set, returns `{ valid: false, error: 'CRON_SECRET not configured', status: 500 }` 
   - Gets `authorization` header from request
   - Uses HMAC-then-timingSafeEqual pattern per CRON-SEC-01:
     ```typescript
     import crypto from 'node:crypto';
     
     function timingSafeCompare(a: string, b: string): boolean {
       const key = crypto.randomBytes(32);
       const hmacA = crypto.createHmac('sha256', key).update(a).digest();
       const hmacB = crypto.createHmac('sha256', key).update(b).digest();
       return crypto.timingSafeEqual(hmacA, hmacB);
     }
     ```
   - Returns a result object: `{ valid: true }` or `{ valid: false, error: string, status: number }`
   - Type the return: `type CronAuthResult = { valid: true } | { valid: false; error: string; status: number }`

2. `getRedis(): Redis | null` — Per CRON-CLEAN-01, lazy singleton.
   - Module-level `let _redis: Redis | null | undefined;` variable
   - On first call: reads `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` from `process.env`
   - If either is missing, caches and returns `null`
   - Otherwise creates `new Redis({ url, token })`, caches in `_redis`, returns it
   - Subsequent calls return cached instance (no new Redis() per call)

Do NOT add `import 'server-only'` — this module is used by API route handlers which are not server components.
