    Create systematic RBAC tests for server actions using describe.each pattern from RESEARCH.md. The role hierarchy is: user < staff < manager < admin < super_admin.

    ```typescript
    const ROLES = ['user', 'staff', 'manager', 'admin', 'super_admin'] as const;
    ```

    For each category of actions, test that:
    1. Roles BELOW the minimum required role are REJECTED (returns { success: false } or throws Unauthorized)
    2. Roles AT or ABOVE the minimum required role SUCCEED (returns { success: true } when DAL is mocked)
    3. Unauthenticated (null session) is always REJECTED

    **Actions to test by minimum role:**
    - **admin-only:** createProductAction, deleteProductAction, approveDesignAction, rejectDesignAction, createMediaAction, deleteMediaAction (read each file to confirm minimum role)

    - **user-level (portal):** signConsentAction, updateProfileAction (portal-actions)

    Use describe.each to parameterize the role matrix. For each action:
    ```typescript
    describe.each([
      { action: 'createProductAction', minRole: 'admin', file: '@/lib/actions/product-actions' },

    ])('$action requires $minRole', ({ action, minRole, file }) => { ... });
    ```

    Create integration-style E2E tests that exercise multi-step business flows by calling actions and routes in sequence. These are NOT browser-level E2E tests -- they use the same vitest + vi.mock infrastructure but test the sequence of operations a user would perform.

    **Flow 1: Guest Checkout (store purchase without login)**
    1. Mock store product data in DAL
    2. Call store checkout action/route with cart items and payment info
    3. Simulate Stripe webhook (payment_intent.succeeded) by calling the webhook route handler with a mocked event
    4. Verify order was created (mock DAL should have been called with correct order data)
    5. Verify download token was generated (for digital products)

    **Flow 2: Tattoo Session Payment**
    1. Auth as admin
    2. Call createCustomerAction to create a customer
    3. Call session creation action to create a tattoo session
    4. Call payment request action to send payment request
    5. Simulate Stripe webhook for payment received
    6. Verify session status updated and payment linked

    **Flow 3: Portal Consent Signing**
    1. Auth as user (portal user)
    2. Mock customer linkage (user linked to customer record)
    3. Call portal sessions endpoint to fetch sessions requiring consent
    4. Call signConsentAction with consent form data
    5. Verify consent recorded (mock DAL called with correct consent data)
    6. Verify session now shows consent as signed

    **Flow 4: Admin CRUD Flow**
    1. Auth as admin
    2. Call createCustomerAction with valid customer data -> verify returns { success: true, data: { id } }
    3. Call updateCustomerAction with updated data -> verify returns { success: true }
    4. Call deleteCustomerAction with the created ID -> verify returns { success: true }
    5. Verify correct DAL calls were made in order (create, update, delete)

    **Implementation approach:**
    - Each flow is a describe() block with a sequential set of it() blocks
    - Use shared mock state across tests in a flow (e.g., the customer ID from create is used in update/delete)
    - Mock ALL external dependencies (DB, Stripe, Resend, Vercel Blob)
    - Focus on verifying the correct sequence of DAL calls and return values
    - Use assertions on mock call arguments to verify data flows correctly between steps

    **CRITICAL:** These tests verify the integration between actions/routes, NOT the individual action logic (that is covered by TEST-01 and TEST-02 tests). Focus on data flow between steps and correct sequencing.
