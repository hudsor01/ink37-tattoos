Upgrade all 4 dashboard forms. DO NOT rewrite forms from react-hook-form to useActionState. Keep RHF forms intact. Add these enhancements:

**Clarification on D-07 (addresses review concern):** CONTEXT.md D-07 mentions useActionState, but ALL existing dashboard forms use react-hook-form (RHF) with zodResolver. Research confirmed that rewriting to useActionState would be an anti-pattern -- it would lose client-side instant validation and create unnecessary churn. The correct approach per research is: keep RHF for client-side validation, and use `form.setError()` to map server-side `ActionResult.fieldErrors` onto the same RHF form fields. The existing shadcn `FormMessage` component already renders RHF errors below inputs, so server errors display automatically through the same mechanism. This is the RHF-integrated approach that both reviewers endorsed.

**A. Server-side field error display (UI-06, per D-07 intent -- using RHF form.setError)**

In each form's `onSubmit` handler, after calling the server action:

```tsx
const result = await serverAction(data);
if (!result.success) {
  if (result.fieldErrors) {
    Object.entries(result.fieldErrors).forEach(([field, messages]) => {
      form.setError(field as keyof FormData, {
        type: 'server',
        message: messages[0],
      });
    });
  } else {
    toast.error(result.error || "Something went wrong");
  }
  return;
}
```

The existing `FormMessage` component (from shadcn Form) already renders `form.formState.errors[field].message` below each input. By using `form.setError()`, server errors will automatically display in the existing FormMessage slots. No need to add a separate FieldError component to RHF forms -- the FormMessage handles it.

For forms that do NOT use the shadcn Form pattern (check session-form.tsx which uses raw `register()`), add `<FieldError>` components from Plan 01 below each input, passing the error from `form.formState.errors[fieldName]?.message`. Wrap the error message in an array for FieldError: `errors={error ? [error.message ?? ''] : undefined}`.

**B. DatePicker for date inputs (UI-08, per D-10)**

1. **customer-form.tsx**: Replace `<Input type="date">` for `dateOfBirth` with:

   ```tsx
   <FormField
   ```

```tsx
toast.promise(

    // success handling
    return result;

  }
);
```

If a form uses a different pattern (bare toast.success/toast.error), migrate to toast.promise for consistency. If toast.promise is already used, leave it.
