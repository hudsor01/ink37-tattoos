  <action>
Create `.github/workflows/ci.yml` with the following configuration:

**Trigger:** On pull_request targeting main, and on push to main (so main branch always has a green status).

**Environment:**
- runs-on: ubuntu-latest
- Use oven-sh/setup-bun@v2 to install bun (do NOT use npm/node setup)
- Bun version: latest

**Jobs -- single job called "ci" with these steps:**
1. Checkout code: actions/checkout@v4
2. Setup Bun: oven-sh/setup-bun@v2
3. Install dependencies: `bun install --frozen-lockfile`
4. Run tests: `bun run test` (this runs `vitest run --reporter=verbose`)
5. Run build: `bun run build` (this runs `next build`)

**Environment variables needed for build (Next.js needs these at build time):**
The build will need database and auth env vars to compile. Use GitHub Actions environment secrets. Add these as env vars on the build step, pulling from GitHub secrets:
- DATABASE_URL: ${{ secrets.DATABASE_URL }}
- BETTER_AUTH_SECRET: ${{ secrets.BETTER_AUTH_SECRET }}
- BETTER_AUTH_URL: ${{ secrets.BETTER_AUTH_URL }}
- NEXT_PUBLIC_APP_URL: ${{ secrets.NEXT_PUBLIC_APP_URL }}

For the test step, tests use mocked dependencies so no real env vars are needed. Set dummy values:
- DATABASE_URL: "postgresql://test:test@localhost:5432/test"
- BETTER_AUTH_SECRET: "test-secret-key-for-ci"
- BETTER_AUTH_URL: "http://localhost:3000"
- NEXT_PUBLIC_APP_URL: "http://localhost:3000"

**Important details:**
- Do NOT use npm or yarn anywhere -- bun only
- Use `--frozen-lockfile` flag on `bun install` so CI fails if lockfile is out of sync
- The lockfile is `bun.lock` (text format, not binary bun.lockb)

After creating the file, commit it to main and push:
```
git add .github/workflows/ci.yml
git commit -m "ci: add GitHub Actions workflow for test and build on PRs"
git push origin main
```

Then verify the workflow registered:

gh workflow list

<task type="auto">
  <name>Task 2: Enable branch protection and verify Vercel integration</name>
  <read_first>
    - Run `gh repo view --json owner,name --jq '"\(.owner.login)/\(.name)"'` to get owner/repo
    - Run `gh workflow list` to get the workflow name for branch protection rule
    - Run `gh api repos/{owner}/{repo}/branches/main/protection 2>/dev/null` to check existing protection
  </read_first>
  <files></files>
  <action>
1. **Enable branch protection on main** requiring the CI check to pass before merge:

   Use the GitHub API via gh CLI to set branch protection:
   ```
   gh api repos/{owner}/{repo}/branches/main/protection \
     --method PUT \
     --field required_status_checks='{"strict":true,"contexts":["ci"]}' \
     --field enforce_admins=false \
     --field required_pull_request_reviews=null \
     --field restrictions=null
   ```

   Notes:
   - `strict: true` means the branch must be up to date with main before merging
   - `contexts: ["ci"]` references the job name from the workflow file
   - `enforce_admins: false` so the solo developer (admin) is not blocked
   - No required reviewers (solo developer)
   - No restrictions on who can push

   Get owner/repo from: `gh repo view --json owner,name --jq '"\(.owner.login)/\(.name)"'`

2. **Verify Vercel GitHub integration is active:**

   Check if Vercel is connected to this GitHub repo:
   ```
   gh api repos/{owner}/{repo}/hooks --jq '.[].config.url' 2>/dev/null
   ```

   Also check for Vercel bot deployments in recent activity:
   ```
   gh api repos/{owner}/{repo}/deployments --jq '.[0:3] | .[].environment' 2>/dev/null
   ```

   If Vercel integration is NOT connected (no hooks or deployments visible), note this as a finding. The user will need to:
   - Go to vercel.com/dashboard
   - Select the ink37-tattoos project
   - Go to Settings > Git
   - Connect to the GitHub repository
   - Set Production Branch to "main"

   If Vercel integration IS connected, verify it targets main as the production branch.

3. **Test the CI pipeline end-to-end** by creating a test PR:
   ```
   git checkout -b ci/test-workflow
   ```
   Make a trivial change (add a comment to .github/workflows/ci.yml or create a small test file).
   ```
   git add . && git commit -m "ci: test workflow trigger"
   git push origin ci/test-workflow
   gh pr create --base main --head ci/test-workflow --title "CI Test: verify workflow triggers" --body "Testing that CI workflow runs on PR. Will be closed after verification."
   ```

   Wait for the workflow to start:
   ```
   gh run list --branch ci/test-workflow --limit 1
   ```

   If the run started, the CI workflow is working. Close and clean up the test PR:
   ```
   gh pr close --delete-branch
   ```
   Then locally: `git checkout main`

   If the run did NOT start, troubleshoot: check `gh workflow list`, check if the workflow file syntax is valid with `gh workflow view ci.yml`.
  </action>
